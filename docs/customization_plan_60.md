## A. 品牌化与命名空间（1-10）

1. 建立统一品牌词典（项目名、简称、产品线代号、Logo 规范）。
2. 把品牌配置从文档固化到机器可读文件（YAML/TOML）。
3. 对 README、CLI Banner、API 文案做统一替换校验。
4. 扫描代码中的 `unsloth` 字符串并标注“必须保留/可替换”。
5. 为 Java 子模块预留 `groupId/artifactId/package` 批量替换规则。
6. 为 Python 包预留分发名与导入名解耦策略。
7. 设定仓库命名约束：目录、镜像名、发布包名一致。
8. 增加品牌一致性 CI 检查，防止新提交回写旧品牌名。
9. 将 Logo/封面图纳入 `docs/assets` 并版本化管理。
10. 在发布流程中加入“品牌检查门禁”。

## B. 配置治理与敏感信息（11-20）

11. 建立 `.env.example`、`.env.dev`、`.env.prod` 多环境模板。
12. 数据库、Redis、Ollama、OpenAI 地址统一走环境变量。
13. API Key 一律从 Secret Manager 注入，不落盘。
14. 对配置值增加启动时校验，避免空值上线。
15. 对 URL 做协议与主机白名单验证。
16. 把配置读取逻辑集中在一个模块，禁止散落式 `os.getenv`。
17. 提供“配置健康检查”接口，便于部署前校验。
18. 在日志中对密钥与令牌做脱敏打印。
19. 为关键配置提供热更新/重载能力。
20. 配置变更必须留审计日志（谁、何时、改了什么）。

## C. 依赖升级与供应链（21-30）

21. 制定“依赖分层升级”策略：运行时 > 编译时 > 开发时。
22. 每周自动生成 `outdated` 报告并归档。
23. 对高风险依赖建立 CVE 预警机制。
24. 对 Python 和 Node 依赖做许可证扫描。
25. 关键依赖升级前先跑兼容性冒烟测试。
26. 对前端大版本升级单独开分支验证。
27. 为 GPU 相关依赖建立硬件矩阵验证。
28. 设置依赖锁文件策略，减少“同版本不同结果”。
29. 建立回滚清单：升级失败时可一键回退。
30. 把依赖升级记录写入 `CHANGELOG`。

## D. 代码重构与架构优化（31-40）

31. 把业务配置、模型配置、系统配置分层隔离。
32. 将训练、推理、导出等模块解耦为独立服务边界。
33. 抽离统一错误码与异常处理层。
34. 引入服务接口层，减少控制器直连底层实现。
35. 对长函数拆分为“校验-编排-执行-回写”四段。
36. 对状态共享改为显式上下文对象，避免隐式全局变量。
37. 为任务编排增加可观测事件总线。
38. 对 I/O 密集逻辑引入异步队列与背压控制。
39. 关键路径加缓存层并定义失效策略。
40. 给插件化能力定义稳定 SPI（扩展点协议）。
